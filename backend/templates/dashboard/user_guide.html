Operations Manual

User Guide

This guide explains what each module does, why it exists, and how to move through the full event workflow safely. It is designed for both first-time users and experienced users, so you can use it as a quick reference or as a complete handover guide.

Work stage by stage. If something looks wrong in a later module, first verify the earlier module instead of forcing a correction downstream.

Best Way To Use This App

Enter source data carefully, sync each later stage from the previous one, reconcile before moving forward, and export only when the current stage looks clean.

Common Button Meanings

  • Sync Data: pull data from the previous stage.
  • Upload CSV: intentionally replace the working copy from an external file.
  • Save as Draft: keep work incomplete and editable.
  • Submit: move that stage forward for downstream use.
  • Export: download the current stage output.

1Before You Start

The app works in stages. Later modules depend on earlier modules, so correct data should always be fixed at the source stage first.

Draft Use when the information is incomplete and still under review.
Submit Use only when that stage is ready to influence later modules.
Sync Use to pull the latest approved data from the previous stage.
1
Enter clean source data

Application Entry is the source truth. If data is wrong there, later modules will simply carry the mistake forward.

2
Sync the next stage

Use `Sync Data` to refresh the current working stage from the previous stage instead of manually recreating rows.

3
Reconcile before moving forward

When a stage has reconciliation, health, or sync checks, review them before continuing to the next module.

2Base Files

Base Files are the backbone of the yearly workflow. Upload these first, because the rest of the app depends on this reference data.

What Lives Here

  • District Master: the yearly district reference with application number, district name, president name, mobile number, and allotted budget.
  • Article Price List: the article master used for article name, cost, item type, category, super category, and token name.
  • Past History: previous-year beneficiary history used for operational reference and verification during current entry.

Why It Comes First

  • Base Files are the curated starting point for the year.
  • District master is used to create district applications with the correct district-side details already available.
  • Article price list gives the default article information used across Application Entry and later stages.
  • Past history helps users verify whether a beneficiary has already appeared in previous years.
Working order: Base Files first, then Application Entry, then the later operational stages. If Base Files are outdated, the rest of the process becomes harder to trust.

3Dashboard

The Dashboard is the first operational summary of the app. It does not create records, but it helps the team judge whether the current numbers look believable before investigating a module.

What It Summarizes

  • Application counts across District, Public, and Institutions.
  • Beneficiary split information such as handicapped and gender-based views.
  • Fund request and operational totals coming from later modules.
  • Pending district indicators so the team can see which districts still need work.

How To Use It

  • Check totals before and after major changes or imports.
  • If a number feels wrong, trace it back to the source module instead of correcting the dashboard.
  • Use it as a health indicator and overview, not as a data-entry screen.

4Application Entry

This is the source truth for beneficiaries, requested items, quantities, costs, and aid-specific details.

▾

What You Do Here

  • Create and maintain District, Public, and Institution applications as the source truth for the rest of the app.
  • District applications can carry multiple requested items. Public applications carry one article or aid in a single application. Institution applications carry institution-level entries and item details.
  • The expanded overview is a quick operational summary. The eye view is the full detail screen for reviewing every saved field and item row.

Important Working Logic

  • Aadhaar verification: for public entry, the verify action checks past history and current public records for the same Aadhaar so users can identify returning beneficiaries or current duplicates before proceeding.
  • Current duplicate protection: if the same Aadhaar already exists in current public records, the public form locks and points the user to modify the existing record instead of creating a duplicate one.
  • Handicapped and gender category: these are collected for beneficiary classification, reporting, and operational categorization used later in distribution and analysis.
  • Internal notes and comments: district and institution forms keep office-side internal notes; public form comments help record communication or special context about that beneficiary.

Article, Cost, and Total Logic

  • Cost per unit is usually auto-filled from the Article Price List.
  • If the stored article cost is zero or not yet finalized, the user can continue entry without stopping the whole process.
  • Total value is derived from quantity and cost per unit.
  • This keeps data entry moving, but the cost and total value must be reviewed before later operational stages depend on them.

Duplicates and Multiple Rows

  • Multiple aid rows can exist when they represent genuinely different aid entries for the same beneficiary.
  • For non-aid article rows, the app warns against repeated same-item rows because they can create duplicate operational records later.
  • If a required article is missing, users can add a new article during entry itself instead of always leaving the screen to go to Article Management.

Public Form Specifics

  • Public entry is intentionally single-entry: one beneficiary can have one article or one aid in a single public application.
  • Disability category becomes relevant only when handicapped is marked.
  • Gender category is selected separately to help later classification and reporting.

District and Institution Specifics

  • District entry starts from District Master and shows district code, president, mobile, allotted budget, and remaining fund.
  • Institution application number is generated on submit rather than entered first.
  • Both district and institution forms support multiple item rows and reveal the extra aid fields only when the selected item type is Aid.
Important: If an application is saved under the wrong district, wrong beneficiary, or wrong item, later modules will inherit that mistake. Fix it here first. Save as draft when unsure, and submit only when the source record is ready for downstream use.

5Article Management

Maintain the master list of articles that can be used across the app.

▾

What This Module Maintains

  • Article name
  • Token Name
  • Cost per unit
  • Item type
  • Category helps you group articles for reporting.
  • Super Category helps higher-level reporting and summary views.
  • These fields make later exports, dashboards, planning views, and audits easier to read and summarize.

What Token Name Does

  • Token Name is the shortened article name intended for tokens and labels.
  • It is prepared here early so users do not need to shorten labels at the last minute during token and label generation.
  • Use a readable short version that still clearly identifies the item.
  • The list page supports search, export, and filtering by item type, category, super category, and combo / separate state.
Tip: Create or clean articles here when you want a stable reusable master record. For urgent cases, an article can still be introduced from Application Entry and then normalized later if needed.

6Inventory Planning

See the item-level requirement and planning state before distribution and ordering actions.

▾

How The Main Numbers Work

  • Needed is the required quantity coming from application-side demand.
  • Ordered is what later ordering actions have already moved forward.
  • Pending is the remaining quantity still to be handled.
  • Excess appears when ordered quantity goes beyond the needed quantity.

Filters and Breakdown

  • The top metric cards summarize Article, Aid, and Project separately.
  • Filters include item type, application status, combo, order status, balance, and search.
  • Expanding a row shows the beneficiary-level breakdown behind that order item, including application status and order status.
Operational meaning: this module is not placing orders itself. It is the planning and monitoring layer that shows what is still needed and how far ordering activity has already covered the demand.

7Order & Fund Request

Used for both Aid requests and Article requests, with different data-entry behavior for each type.

▾

Aid Request

  • Select the aid type first.
  • Choose the beneficiaries and confirm the request details.
  • Use draft when the beneficiary or amount details are still being checked.
  • The beneficiary list is pulled from the relevant application data for the selected aid type, and the request copies those details into the fund request working form.
  • The detail page clearly shows whether the request is still draft or has already moved quantities into the ordered state.

Article Request

  • Group items under vendors.
  • Assign the required items vendor-wise.
  • The vendor grouping is meant to avoid repeating the same vendor details for every article row.
  • Submit only after the request is ready to update the inventory-side ordered values and move the operational status forward.
  • The list page separates drafts from submitted records, and only submitted requests move mapped quantities into the ordered state shown in Inventory Planning.
Important: Aid requests and Article requests are not the same workflow. Aid requests copy beneficiary-side details from application data, while article requests group vendor and item ordering details. Confirm the request type before adding rows, because the draft or submitted state has an inventory effect.

8Seat Allocation

Split each row into Waiting Hall and Token quantities while keeping the total quantity correct.

▾
A
What Sync Data does

`Sync Data` pulls the latest Master Data into the Seat Allocation working copy. It refreshes the stage copy only; it does not edit Application Entry directly.

B
What Upload CSV does

`Upload CSV` lets you intentionally replace the Seat Allocation working copy with an external file.

C
How to work here

Check the source summary and reconciliation first, use filters to focus on the right rows, then decide how many beneficiaries collect in Waiting Hall and how many on stage through Token quantity.

D
Bulk action and save

Bulk action is kept to speed up repeated splits across many similar rows. After editing, use save and review the reconciliation so you know Waiting Hall plus Token still matches the original total quantity from the source copy.

Meaning of the split: Waiting Hall and Token are not two different source rows. They are the operational split of one source requirement. The stage works only if the split total still equals the original quantity.

9Sequence List

Assign one sequence number to each item so distribution, tokens, and labels follow a fixed order.

▾
  • `Sync Data` pulls the current item list from the earlier stage into the Sequence working copy.
  • Move items from left to right and arrange them in the required order.
  • `Save and Reconcile` saves the sequence list and checks that the final sequence output still matches the earlier master and seat stages before it is treated as trustworthy.
  • The logic here is item-level: one item should map to one sequence number, and one sequence number should map to one item order.
  • Use export when the final sequence output is ready.
Important: If Sequence reconciliation shows a mismatch, do not use Sequence List to hide the issue. Fix the upstream stage first. Sequence is the trust layer before token generation, so its reconciliation is meant to protect the final operational file.

10Token Generation

Prepare the final working file for token-based distribution, labels, and downstream reporting.

▾
1
Sync or Upload

Load the current sequence-stage file into Token Generation. Sync pulls from the previous app stage; upload intentionally replaces the token working copy from an external file.

2
Data Prep

Create the working `Names` field, complete missing values where needed, sort the data into the required order, and run quality checks before token generation continues.

3
Adjust Names

Shorten long `Names` and `Token Name` values so they fit correctly in labels and reports. The page shows only the unique long values so the user can review them efficiently.

4
Token Print Review

Decide which article rows should skip article-label sticking, because not every token-bearing row needs a physical article sticker.

5
Generate Token

Create `Start Token No` and `End Token No` ranges from the prepared token quantity, then export the final token file.

Why this stage exists: Token Generation turns the reconciled sequence output into the final working file used for token-based reporting, badge creation, and label generation.

11Labels

Generate printable PDF labels from the final token-generated data.

▾
  • `Sync Data` pulls the current token-generated file into the Labels stage.
  • `Upload` lets you load a prepared label file manually.
  • Article labels are used to stick on physical items.
  • District, Institution, and Public labels act as badges or token labels for the relevant beneficiaries.
  • Article labels support multiple layouts like 12L and 2L, while other label groups provide the relevant continuous or separate badge sheets.
  • Chair labels and custom labels are available for special operating needs.
Working rule: Labels should be generated from token-generated data, not from earlier stages. That is what keeps token number ranges, names, and token names aligned with the final distribution file.

12Purchase Order

A separate workflow from Fund Request, used for vendor-facing purchase documents.

▾
  • Purchase Order is independent from Fund Request and works directly with vendor and ordered-item details.
  • Add vendor details, GST number, and ordered items.
  • Each item row carries article, quantity, unit price, line total, supplier article name, and description.
  • Use drafts while preparing the document.
  • Submit only when the purchase order is finalized, then generate the printable PDF.

13Reconciliation & Admin

Supporting modules that help you maintain trust, history, access control, and operational confidence across the app.

▾

Reconciliation

  • Reconciliation: run live data and app health checks across modules.
  • Use it when you want confidence that dashboard values, source truth, seat allocation, sequence, exports, and key regressions still align.
  • Data Health checks current live values. App Health runs regression suites safely in a test database so live data is not changed.

Audit Logs & User Management

  • Audit Logs: see who changed what and when.
  • User Management: create users, manage roles, and reset passwords.
Support tip: If something feels wrong, first check this User Guide, then the Reconciliation page, and then Audit Logs to understand what changed.
